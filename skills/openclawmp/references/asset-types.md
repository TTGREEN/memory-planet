# OpenClaw 水产市场资产类型说明

当用户需要理解资产分类差异、发布前难以判断 `assetType`、或某个资产看起来同时具备多种属性时，读取本文件。

本文件是 `SKILL.md` 中“资产类型速判”的展开说明。若本文件与真实平台规则冲突，以平台最新规则和 CLI/服务端实际行为为准。

## 五种资产类型

### `skill`

定义：

- Agent 可直接学习和复用的能力包
- 通常包含提示词、执行步骤、脚本、参考资料或流程约束
- 核心价值是告诉 agent “怎么完成某类任务”

典型例子：

- 代码审查流程
- 天气查询流程
- 内容写作流程
- 某类分析任务的标准操作步骤

判断重点：

- 是否在教 agent 一个可重复执行的方法
- 是否以“任务流程”而不是“底层工具接入”为核心

### `plugin`

定义：

- 为 agent 提供代码级扩展能力
- 核心价值是接入新的工具、API、服务或 MCP 能力
- Agent 通常通过 tool call 或程序接口使用它

典型例子：

- GitHub API 工具接入
- Yahoo Finance API 接入
- MCP server 封装
- 第三方服务连接器

判断重点：

- 是否主要在提供一个“能调用的新能力”
- 是否更像工具层、接入层，而不是任务方法论

### `trigger`

定义：

- 负责监听事件或按时间计划调度 agent
- 核心价值是在特定条件满足时唤起 agent
- 通常是单向触发，不强调对外持续双向通信

典型例子：

- 文件变更监控
- Webhook 事件触发
- 定时任务
- cron 驱动的自动执行器

判断重点：

- 是否由事件或计划驱动
- 是否核心作用是“触发 agent 开始做事”

### `channel`

定义：

- 负责将 agent 接入一个消息渠道或通信平台
- 核心价值是让 agent 能收消息、发消息、参与会话
- 强调双向通信

典型例子：

- 飞书适配器
- Telegram bot 接入
- 其他 IM、消息或聊天平台接入

判断重点：

- 是否同时处理消息接收与消息发送
- 是否核心作用是“接入一个对话/消息通道”

### `experience`

定义：

- 沉淀实践方案、配置思路、模板、策略或资产组合包
- 核心价值是提供经验、方案和组织方式
- 往往不是一个可严格重复执行的单一任务流程

典型例子：

- 三层记忆系统方案
- 人格模板
- 一套资产组合建议
- 某类系统的落地经验总结

判断重点：

- 是否更像“经验沉淀”而不是“步骤化流程”
- 是否主要帮助用户理解如何配置、组合和设计，而非直接执行一个具体任务

## 常见混淆与区分规则

### `skill` vs `plugin`

区分原则：

- `skill` 解决“怎么做”
- `plugin` 解决“能不能调这个工具/服务”

可用判断：

- 如果主要内容是提示词、步骤、判断规则、脚本工作流，归 `skill`
- 如果主要内容是 API 接入、工具暴露、函数调用能力，归 `plugin`

示例：

- 教 agent 怎么做代码审查：`skill`
- 给 agent 一个可调用的 GitHub API 工具：`plugin`

### `skill` vs `experience`

区分原则：

- `skill` 强调重复执行
- `experience` 强调经验沉淀与方案设计

可用判断：

- 如果 agent 会反复按该流程完成任务，优先归 `skill`
- 如果内容更像最佳实践、架构建议、模板方案，优先归 `experience`

示例：

- 教 agent 写小红书文案的完整执行流程：`skill`
- 分享“三层记忆系统如何搭建”的方案：`experience`

### `trigger` vs `channel`

区分原则：

- `trigger` 是“发现条件成立，然后叫醒 agent”
- `channel` 是“把 agent 接进一个双向消息系统”

可用判断：

- 如果是事件监听、定时执行、Webhook 驱动，优先归 `trigger`
- 如果是消息平台适配、群聊收发、机器人接入，优先归 `channel`

示例：

- 监控某目录有新文件就触发分析：`trigger`
- 飞书群里收消息并回消息：`channel`

### `trigger` vs `skill`

区分原则：

- `trigger` 看“谁启动”
- `skill` 看“启动后做什么”

可用判断：

- 如果核心价值是自动唤起机制，归 `trigger`
- 如果核心价值是被调用后的执行流程，归 `skill`

示例：

- 每天 8 点自动生成新闻摘要：触发部分是 `trigger`
- 生成新闻摘要的分析和写作流程：执行部分是 `skill`

## 发布时如何帮助用户选择 `assetType`

当用户准备发布资产但不知道该填什么类型时，按以下顺序判断：

1. 先问“这个资产最核心的价值是什么？”
2. 只按核心用途判断，不按实现复杂度判断
3. 如果一个资产同时具备多种属性，优先选择对外暴露的主角色

优先判断规则：

- 核心是任务执行流程：`skill`
- 核心是新工具或服务接入：`plugin`
- 核心是事件监听或定时唤起：`trigger`
- 核心是消息平台接入与双向通信：`channel`
- 核心是方案、模板、经验或组合建议：`experience`

## 上传前的 README 要求

在检查或发布资产时，不再强调不同资产类型使用不同的人类说明文件；统一要求所有资产在上传前都提供根目录 `README.md`。

`README.md` 至少要满足以下要求：

- 必须有一级标题，即第一层标题形如 `# 标题`
- 必须有描述段落，即标题后的第一段正文，用来说明该资产解决什么问题、如何使用

推荐最小结构：

```md
# 经验标题

这是描述段落，说明这个经验、触发器、插件或通道解决什么问题、如何使用。

## 安装

...

## 配置

...
```

如果缺少一级标题，或标题后的第一段不是描述段落，而是列表、代码块、空白占位或次级标题，应视为结构不完整。

推荐原因：

- 便于用户快速理解资产用途
- 便于展示安装方式和配置方式
- 便于平台页面、仓库浏览和人工维护
- 便于在 agent 之外的人类场景中快速阅读和评估资产

## 关于 `experience` 的使用边界

`experience` 可以作为较宽松的经验类资产类型，但不要把它当成默认垃圾桶。

只有在以下情况下优先考虑 `experience`：

- 内容主要是方案、策略、模板或经验总结
- 它不适合作为严格步骤化的重复执行流程
- 它也不是工具接入、触发器或消息通道

如果只是因为“暂时不确定”，不要立刻选 `experience`；应先澄清资产的核心用途。

## 给 agent 的实际使用建议

在以下场景读取并应用本文件：

- 用户直接询问五种资产类型的区别
- 用户发布资产但不确定 `.metadata.json` 中的 `assetType`
- 某个资产描述同时包含工作流、工具接入和触发逻辑，导致类型边界模糊

回复时：

- 先给结论，再给理由
- 优先使用简短区分规则
- 只有用户继续追问时，再展开详细例子
